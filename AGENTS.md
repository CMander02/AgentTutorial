# AgentTutorial 项目说明

本项目是一个 MkDocs Material 教程站点，面向首次系统学习 Agent 和 Coding CLI 的读者。所有文档改动都应优先保证新手能按顺序读懂、能自己验证、能知道下一步做什么。

## 常用命令

```powershell
uv run mkdocs serve
uv run mkdocs build --strict
npm run check
npm run fix
```

## 格式规范

本项目参考 [LUG@USTC Linux 101](https://github.com/ustclug/Linux101-docs) 的 Prettier 配置：

```json
{
    "embeddedLanguageFormatting": "off",
    "tabWidth": 4
}
```

仓库根目录提供 `package.json`。修改 Markdown 后，先运行 `npm run fix`，再运行 `npm run check` 和 `uv run mkdocs build --strict`。

## 页面结构

每个正文页面优先使用以下结构。内容很短时可以合并相邻小节，但读者仍应看得出目标、步骤和验证方式。

1. H1 标题。
2. `!!! abstract "导言"`：只用于导言，默认展开。
3. `## 目标 {#goal}`：说明读者完成后能做什么。
4. `## 背景 {#background}`：补足必要前置知识。
5. `## 步骤 {#steps}`：给出可执行路径。
6. `## 验证 {#check}`：列出读者怎样确认已经做对。
7. `## 常见误区 {#pitfalls}`：收纳风险、边界和容易误解之处。
8. `## 下一步 {#next}`：指向导航里的下一页。

## Markdown 写作规则

- 每页只有一个 H1，H1 不加 `{#id}`。
- H2 及以下标题必须添加稳定 ID，例如 `## 验证 {#check}`。
- ID 使用小写字母、数字和短横线；避免中文、空格和大写。
- 所有提示框标题行后保留一个空行。
- 只有导言使用 `!!! abstract "导言"`。
- 其他块状说明统一使用 `???` 折叠块，例如 `??? info "重点"`、`??? warning "注意"`。
- Prompt、命令、代码、配置和模板必须放在普通 fenced code block 中，便于复制；不要放进提示框。
- 用提示框解释风险或补充背景，不要用提示框承载主要操作步骤。
- 图片需要配字时，图片下方紧接一行说明，再紧接 `{: .caption }`。

## 中文行文规则

- 第 3 章之后的页面应避免翻译腔句式，尤其避免反复使用“不是……而是……”“是……不是……”“……而非……”“而不是……”。
- 避免用逗号硬连接两个短分句。需要表达因果、条件、递进或并列时，补上“因此”“当……时”“如果……就”“并”“以及”“从而”等关系词。
- 购买、价格、模型名称和产品额度经常变化。正文应写判断标准和验证清单，少写短期推荐。
- 新手教程要避免内部黑话。首次出现 `MCP`、`Skill`、`AGENTS.md` 等概念时，应给一句用途解释。

## Linux 101 写作规范在本项目中的落地

Linux 101 的完整原文规范保存在 `.codex/skills/linux101-writing/references/writing.md`。本项目采用其中这些核心规则：

- 章节开头写导言，帮助读者建立前置知识和内容地图。
- 主体从 H2 开始组织，H2、H3 各自承担独立模块。
- 正文段落要有完整逻辑，避免知识点依赖倒置。
- 标题手动添加 ID，保证内部链接长期稳定。
- 善用提示框区分重点、范例、小知识和警告。
- 引用、脚注和外链应服务正文理解，不能打断阅读主线。
- 本地预览和格式检查要成为每次文档修改后的固定动作。
