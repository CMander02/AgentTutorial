# Prompt 与计划

!!! abstract "导言"

    很多 Agent 任务失败，常见原因是请求里缺少目标、边界、资料和验收标准。写清 Prompt，就是把任务合同写清楚；要求计划，就是在改文件前先检查路线。

## 目标 {#goal}

本章让你学会三件事：

- 写出带目标、边界和验收标准的请求。
- 要求 Agent 先计划，再修改。
- 在计划里识别风险和缺口。

## Prompt 六要素 {#six-parts}

一个实用请求通常包含六项：

| 要素 | 说明                 |
| ---- | -------------------- |
| 目标 | 最终要完成什么       |
| 范围 | 可以读写哪些文件     |
| 背景 | 任务为什么存在       |
| 限制 | 哪些事情不能做       |
| 验收 | 怎样判断完成         |
| 输出 | 希望它用什么格式汇报 |

可以使用这个模板：

```text
目标：把 docs/02-choose/ 改成稳定的选型判断，不写短期价格推荐。
范围：只修改 docs/02-choose/ 和 README 中相关链接。
背景：教程面向新手，产品价格和额度经常变化。
限制：不要改 mkdocs.yml，不要删除文件。
验收：运行 uv run mkdocs build --strict 通过。
输出：列出修改文件、验证命令和剩余风险。
```

## 先计划再修改 {#plan-first}

任务稍微复杂时，先要求计划：

```text
请先阅读相关文件，给出修改计划。计划里说明要改哪些文件、每个文件改什么、可能的风险。先不要修改文件。
```

确认计划后，再发起修改：

```text
按刚才的计划执行。保留已有未提交改动，完成后运行格式检查和构建检查。
```

## 看计划的四个点 {#review-plan}

检查计划时，重点看：

- 文件范围是否符合你的要求。
- 是否识别了已有改动。
- 验证命令是否足够。
- 是否存在删除、迁移、联网或账号操作。

??? warning "计划太泛时要追问"

    如果计划只写“优化文档”“调整结构”，需要让 Agent 具体到文件和小节。计划越具体，后续 diff 越容易审。

## 验证 {#check}

完成本章后，你应该能把一句模糊请求改写成明确请求。

模糊请求：

```text
帮我优化一下教程。
```

明确请求：

```text
请只阅读 docs/03-pre-agent/00-index.md，指出对新手不清楚的地方。先给 3 条修改建议，不要改文件。
```

## 常见误区 {#pitfalls}

??? warning "把限制写在最后一句"

    限制条件要醒目，最好单独成段。关键限制可以重复出现在“范围”和“限制”两处。

??? warning "没有验收标准"

    没有验收标准时，Agent 很容易把“写完回答”当成完成。要写清楚命令、页面、diff 或人工检查标准。
