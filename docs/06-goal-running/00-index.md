# 长任务与目标

!!! abstract "导言"

    当任务跨越多个文件、多个验证命令或较长时间时，单条 Prompt 很难承载全部上下文。目标管理的作用，是让 Agent 一直围绕最终结果工作，并在中断后能接着推进。

## 目标 {#goal}

本章让你学会：

- 什么时候使用 `/goal`。
- 如何把长任务拆成检查点。
- 如何处理中断、恢复和交接。

## 适合使用目标的任务 {#when}

这些任务适合声明目标：

- 重组文档结构。
- 连续修复构建错误。
- 批量迁移页面格式。
- 开发一个需要多轮验证的小功能。
- 写一份需要查资料、落文件、再验证的报告。

这些任务通常有共同特征：范围较大、步骤较多、验证需要反复运行。

## 声明目标 {#declare}

目标要写最终状态和验收条件：

```text
/goal 把当前 MkDocs 教程重组为 0 到 9 章结构，新增写作规范和 Prettier 检查，并通过 npm run check 与 uv run mkdocs build --strict。
```

如果工具支持目标追踪，它会在后续工作中持续参考这个目标。

## 拆检查点 {#checkpoints}

长任务应拆成检查点：

```text
请把这个目标拆成 5 到 7 个检查点。每个检查点写明要改的文件、验证方式和可能风险。
```

检查点完成后，要求 Agent 更新状态：

```text
请复盘当前目标：已完成什么、正在处理什么、下一步是什么、有哪些风险。
```

## 中断后恢复 {#resume}

恢复任务时，先让 Agent 重新读现场：

```text
请先查看 git status、当前目标和最近修改文件。总结现场后继续，不要直接改文件。
```

如果你换了工具或换了线程，可以要求交接说明：

```text
请写一份交接说明，包含目标、已改文件、验证命令、失败信息、剩余工作和下一步建议。
```

## 验证 {#check}

长任务完成时，至少需要三类证据：

- Git 状态和 diff。
- 格式、测试或构建命令结果。
- 人工检查过的页面、截图或输出摘要。

## 常见误区 {#pitfalls}

??? warning "目标写成过程"

    目标应描述最终状态。过程可以放进计划和检查点。

??? warning "中断后直接继续修改"

    上下文可能已经变化。恢复时先读现场，再继续执行。
