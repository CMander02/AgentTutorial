# Skill 与 MCP

!!! abstract "导言"

    Skill 和 MCP 经常一起出现，但它们解决的问题不同。Skill 负责告诉 Agent 怎样做事，MCP 负责把外部工具暴露给 Agent。先理解这个分工，后面创建自己的 Skill 才不会偏离方向。

## 目标 {#goal}

本章帮助你分清：

- Skill 的作用。
- MCP 的作用。
- 什么时候先写 Skill。
- 什么时候需要接入 MCP。

## Skill 写方法 {#skill}

Skill 是写给 Agent 的操作手册。它通常包含触发场景、工作流程、文件约定、验证命令和注意事项。

适合写成 Skill 的内容：

- 每次都要重复解释的项目规则。
- 某类文档的固定写法。
- 一套稳定的数据处理流程。
- 使用某个本地工具的步骤。

例如本项目的 `linux101-writing` Skill 就记录了页面结构、提示框规则和验证命令。

## MCP 给工具 {#mcp}

MCP 是一种工具接口。它让 Agent 能调用外部能力，例如读取数据库、访问知识库、操作浏览器、搜索文件或连接业务系统。

适合接入 MCP 的情况：

- 任务需要访问普通文件以外的数据源。
- 工具已有稳定 API。
- 手动复制数据会频繁出错。
- 权限和审计边界已经想清楚。

## 先后顺序 {#order}

新手建议先写 Skill，再考虑 MCP。

原因很简单：如果做事方法还不稳定，接入更多工具只会放大混乱。先用文字规则把流程跑通，再把真正需要的工具接进来。

## 对比表 {#compare}

| 项目     | Skill                         | MCP                          |
| -------- | ----------------------------- | ---------------------------- |
| 解决问题 | 怎样做事                      | 能调用什么                   |
| 主要形态 | Markdown 指南、脚本、参考资料 | 工具服务器、接口、资源       |
| 适合阶段 | 流程沉淀                      | 工具集成                     |
| 风险     | 规则写得过长或过时            | 权限过宽、接口失败、数据泄漏 |

## 验证 {#check}

判断一个需求该走哪条路：

```text
如果问题来自“Agent 不知道该按什么规则做”，先写 Skill。
如果问题来自“Agent 没有办法访问某个系统”，再考虑 MCP。
```

## 常见误区 {#pitfalls}

??? warning "把所有问题都交给 MCP"

    工具接口只能提供能力，不能自动提供好流程。流程需要通过 Skill、项目说明和验证命令约束。

??? warning "Skill 写成百科"

    Skill 要帮助 Agent 做事。过长背景、历史说明和无关链接会占用上下文。
